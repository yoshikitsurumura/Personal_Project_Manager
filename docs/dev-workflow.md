# 開発フロー共有資料

この資料は、`Codex / Claude Code / Gemini / Cursor / Windsurf` の役割分担を固定して、作業の衝突を減らすための共有メモです。Gemini に一部実装を依頼する場合も、この基準に合わせます。

## 役割分担
- `Claude Code`
  - 要件整理
  - 実装順の分解
  - レビュー観点の洗い出し
  - 仕様の言語化
- `Codex`
  - 機能実装
  - 型整理
  - リファクタ
  - テストとビルド修正
- `Gemini`
  - README や設計メモの下書き
  - 比較表や説明資料の作成
  - 低リスクな UI 文言修正
  - 小さく閉じた実装
- `Cursor`
  - バグ再現
  - エラー調査
  - 修正候補の比較
- `Windsurf`
  - コードリーディング
  - 既存実装の理解
  - 補助レビュー

## Gemini に任せると安全な作業
- `docs/` 配下の文書作成
- README の改善
- 既存コンポーネント内の文言改善
- 影響範囲が狭い presentational component の追加
- テストケース案の列挙

## Gemini に任せない方が良い作業
- Firestore データモデル変更
- 認証まわり
- 複数ファイルにまたがる状態管理変更
- Docker / Kubernetes の本番設定変更
- 既存ロジックの大規模リファクタ

## 共有フォーマット
Gemini に依頼するときは、最低限この形式で渡す。

### 1. 目的
- 何を変えたいか

### 2. 変更対象
- 対象ファイル
- 触ってよい範囲
- 触ってはいけない範囲

### 3. 完了条件
- どうなれば完了か

### 4. 確認方法
- どの画面で確認するか
- どのコマンドで確認するか

## 依頼テンプレート
```text
目的:
Project 一覧画面の空状態文言を改善したい

変更対象:
components/empty-state.tsx
app/projects/page.tsx

制約:
ロジックは変えない
スタイルの崩れを出さない
Firestore や auth には触れない

完了条件:
Project が 0 件のときに、初心者にも分かりやすい文言が表示される

確認方法:
npm run build
/projects を開き、Project が 0 件の状態を見る
```
